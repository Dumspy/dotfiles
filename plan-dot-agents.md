# Plan: Extract Agent Configuration to `dot-agents`

## Summary

Create a new external repository named **`dot-agents`** that houses all agent-related artifacts (skills, agents, commands, extensions, themes) with first-class support for **both Nix (Home Manager modules)** and **non-Nix (GNU Stow)** environments. It follows the `Mic92/mics-skills` pattern for Nix packaging.

Non-Nix users consume a **`stow` branch** that is auto-generated by a GitHub Action. This branch vendors all external skills (from flake inputs) into a `home/` directory layout suitable for `stow`.

## Answers to Clarifying Questions

| Question | Answer |
|----------|--------|
| What moves to the repo? | Raw files + Nix lib + Home Manager modules (mics-skills pattern). Per-agent program config (opencode.nix, pi.nix) stays in dotfiles. |
| Global skills path? | `~/.agents/skills/` is the shared global path (supported by both Pi and OpenCode). Agent-specific skills go to `~/.pi/agent/skills/` or `~/.config/opencode/skills/`. Repo handles global skills only. |
| External skill inputs? | New repo consumes them as its own flake inputs. |
| Non-Nix workflow? | GNU Stow via an auto-generated `stow` branch (GitHub Action). |
| Extensions? | Kept in the new repo with Nix build logic. |
| Commands structure? | Global folder + agent-specific folders with subfolders for commands, skills, extensions. |
| Repo name? | `dot-agents` |

## 1. Agent Skill Discovery (from upstream docs)

### Pi (`https://pi.dev/docs/latest/skills`)

Pi loads skills from:
- **Global**: `~/.pi/agent/skills/` and `~/.agents/skills/`
- **Project**: `.pi/skills/` and `.agents/skills/` in cwd and ancestor directories
- Discovery rules:
  - In `~/.pi/agent/skills/` and `.pi/skills/`, direct root `.md` files are discovered as individual skills
  - In all skill locations, directories containing `SKILL.md` are discovered recursively
  - In `~/.agents/skills/` and project `.agents/skills/`, root `.md` files are ignored
- Can use `settings.json` `skills` array to add other harness paths (e.g., `~/.claude/skills`)

### OpenCode (`https://opencode.ai/docs/skills`)

OpenCode searches:
- **Project**: `.opencode/skills/<name>/SKILL.md`
- **Global**: `~/.config/opencode/skills/<name>/SKILL.md`
- **Claude-compatible**: `~/.claude/skills/<name>/SKILL.md` and `.claude/skills/<name>/SKILL.md`
- **Agent-compatible**: `~/.agents/skills/<name>/SKILL.md` and `.agents/skills/<name>/SKILL.md`
- For project-local paths, OpenCode walks up from cwd to git worktree root.

### Key Insight

Both agents support **`~/.agents/skills/`** as a global shared directory. This is our primary target for universal skills. Agent-specific skills target `~/.pi/agent/skills/` or `~/.config/opencode/skills/`.

## 3. Nix Flake Design

### Inputs

The new repo consumes all external skill sources that currently live in dotfiles:

```nix
{
  inputs = {
    nixpkgs.url = "github:NixOS/nixpkgs/nixos-unstable";
    home-manager = {
      url = "github:nix-community/home-manager";
      inputs.nixpkgs.follows = "nixpkgs";
    };
    # External skill sources (moved from dotfiles)
    vercel-agent-skills = { url = "github:vercel-labs/agent-skills"; flake = false; };
    expo-agent-skills = { url = "github:expo/skills"; flake = false; };
    agent-browser = { url = "github:vercel-labs/agent-browser"; flake = false; };
    anthropics-agent-skills = { url = "github:anthropics/skills"; flake = false; };
    dex-agent-skills = { url = "github:dcramer/dex"; flake = false; };
    sentry-skills = { url = "github:getsentry/skills"; flake = false; };
  };
}
```

### Outputs

Expose:
- **`packages.<system>.<skill-name>`** — Each skill as a derivation containing `$out/share/skills/<name>/`
- **`packages.<system>.pi-extensions`** — Built Pi extensions (if any)
- **`homeModules.default`** — Bundle module: `programs.dot-agents` with `skills`, `agents`, `enableAllSkills` options
- **`homeModules.<skill-name>`** — Per-skill modules (like mics-skills) for granular enabling
- **`lib`** — The discovery/bundling functions for reuse by dotfiles if needed

## 4. Nix Home Manager Module (`programs.dot-agents`)

Following the `mics-skills` pattern:

```nix
programs.dot-agents = {
  enable = true;

  # Universal skills -> ~/.agents/skills/ (both Pi and OpenCode read this)
  skills = [ "dependabot-solver" "pr-review-resolver" "frontend-design" ];
  enableAllSkills = true; # or [ "local" "vercel" ]

  # Agent-specific skills
  pi.skills = [ ];
  opencode.skills = [ ];

  # Agents (subagent definitions) -> ~/.config/opencode/agents/
  agents = [ "code-simplifier" "librarian" "oracle" ];

  # Pi extensions (built and symlinked to ~/.pi/agent/extensions/)
  pi.extensions = true;
};
```

### Install Targets

| Source | Destination | Agents |
|--------|-------------|--------|
| `skills/` | `~/.agents/skills/` | Pi + OpenCode |
| `pi/skills/` | `~/.pi/agent/skills/` | Pi only |
| `opencode/skills/` | `~/.config/opencode/skills/` | OpenCode only |
| `agents/` | `~/.config/opencode/agents/` | OpenCode only |
| `opencode/commands/` | `~/.config/opencode/commands/` | OpenCode only |
| `pi/extensions/` (built) | `~/.pi/agent/extensions/` | Pi only |

### Install Strategy

Reuse the existing `link` / `symlink-tree` / `copy-tree` logic from `lib/agent-config.nix`:
- **`link`**: `home.file` individual symlinks (good for debugging).
- **`symlink-tree`** (default): `rsync -a --delete` preserving symlinks into target dirs.
- **`copy-tree`**: `rsync -aL` dereferencing symlinks (for immutable setups).

## 5. Non-Nix Workflow: Auto-Generated `stow` Branch

Since the repo uses **flake inputs** for external skills (vercel, expo, anthropic, etc.), non-Nix users cannot access those skills directly from the `main` branch. A **GitHub Action** solves this by generating a `stow` branch that vendors all skills into a `home/` directory.

### GitHub Action: `.github/workflows/generate-stow-branch.yml`

On every push to `main`:
1. Check out `main`
2. Run a Nix build (or shell script) that resolves all flake inputs and copies skills into a `home/` layout:
   ```
   home/
   ├── .agents/
   │   └── skills/           # universal skills (local + external)
   ├── .pi/
   │   └── agent/
   │       ├── skills/       # pi-specific skills
   │       └── extensions/   # pi extensions (if pre-built, or skip)
   └── .config/
       └── opencode/
           ├── skills/       # opencode-specific skills
           ├── agents/       # agent definitions
           └── commands/     # opencode commands
   ```
3. Commit the `home/` directory to the `stow` branch (orphan branch, force-push).

### Non-Nix User Workflow

```bash
git clone --branch stow https://github.com/<you>/dot-agents.git ~/dot-agents
cd ~/dot-agents
stow home -t ~
```

This symlinks:
- `~/.agents/skills` -> `~/dot-agents/home/.agents/skills/`
- `~/.pi/agent/skills` -> `~/dot-agents/home/.pi/agent/skills/`
- `~/.config/opencode/skills` -> `~/dot-agents/home/.config/opencode/skills/`
- etc.

### Why a branch instead of a folder in main?

- Keeps `main` clean and source-of-truth.
- Avoids duplicating vendored files in git history of `main`.
- Non-Nix users can shallow-clone just the `stow` branch.
- Easy to automate and update.

## 6. Extensions

Pi extensions live in `pi/extensions/`:
- TypeScript source files + `package.json` with dependencies.
- Nix: `buildNpmPackage` derivation in `nix/packages.nix` outputs `$out/lib/pi-extensions/`.
- The HM module symlinks the built output to `~/.pi/agent/extensions/`.
- Dotfiles' `pi.nix` injects `NODE_PATH` via a Pi wrapper that references the built package from `dot-agents`.

OpenCode extensions/plugins (if/when they exist) live in `opencode/extensions/` with analogous packaging.

## 8. Migration Steps (from current dotfiles)

### Phase 1: Create `dot-agents` repo at `/home/nixos/Documents/dot-agents`
1. `git init` in `/home/nixos/Documents/dot-agents`
2. Extract `ai/skills/` -> `/home/nixos/Documents/dot-agents/skills/`
2. Extract `ai/agents/` -> `dot-agents/agents/`
3. Extract `ai/commands/` -> `/home/nixos/Documents/dot-agents/opencode/commands/` (opencode-specific)
4. Extract `lib/agent-config.nix` -> `/home/nixos/Documents/dot-agents/nix/lib.nix`
5. Add external flake inputs (vercel, expo, etc.) to `/home/nixos/Documents/dot-agents/flake.nix`
6. Create `nix/skills.nix` registry mapping skill names to paths/packages
7. Create HM modules (`nix/home-manager.nix`, `nix/home-modules.nix`)
8. Set up GitHub Action for `stow` branch generation

### Phase 2: Update dotfiles to consume `dot-agents`
1. Add `dot-agents` as a flake input in `dotfiles/flake.nix`.
2. Remove external skill inputs from dotfiles (vercel, expo, etc.) — they now live in `dot-agents`.
3. Update `modules/home/agent-config.nix`:
   - Remove local `localSkills` / `localAgents` paths.
   - Remove external source wiring.
   - Import `dot-agents.homeModules.default`.
   - Configure `programs.dot-agents` with desired skills/agents.
4. Keep `modules/home/opencode.nix` and `pi.nix` but:
   - `opencode.nix`: reference commands from `dot-agents` package outputs or keep local references.
   - `pi.nix`: reference built extensions from `dot-agents.packages.<system>.pi-extensions` for `NODE_PATH`.
5. Remove `ai/` directory from dotfiles.
6. Test with `./rebuild.sh`.

## 9. Open Questions / Future Work

- **Pi skill directory configuration**: `~/.pi/agent/skills/` is natively supported. No `settings.json` changes needed for global skills if we use `~/.agents/skills/`. Pi-specific skills go to `~/.pi/agent/skills/`.
- **OpenCode local skills path**: OpenCode natively supports `~/.agents/skills/` as a global path. Also supports `~/.config/opencode/skills/`.
- **Future agents (Claude Code, etc.)**: The `stow` branch can easily add `~/.claude/skills/`. The Nix module can add `skillDirs` entries.
- **Per-project skills**: Out of scope for `dot-agents`. Users create `.agents/skills/` in project roots manually.
