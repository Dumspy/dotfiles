**Rationale:** our `auxera` flake input + overlay is the place we already pin

on our schedule, not when nixpkgs-unstable catches up. nixpkgs already has a
from-source herdr (frozen at 0.7.3), but we don't want to wait on the channel.

**Chosen form — faithful to the remote repo's own tooling (decision §11.8
resolved in favour of (a)):** `pkgs/herdr/package.nix` reproduces herdr's own
upstream `nix/package.nix`, sourcing herdr via `fetchFromGitHub` (reusing
nixpkgs' proven src FOD hash `sha256-Q2yvMs/...` for v0.7.3) and
`callPackage`-ing the **vendored `build.zig.zon.nix`** that ships inside the
herdr source (`${src}/vendor/libghostty-vt/build.zig.zon.nix`, generated by
`zon2nix`). It does NOT use nixpkgs' `zig_0_15.fetchDeps` re-implementation,
matching the instruction "same tooling the remote repo uses".

**Why (a) was also the better choice for maintenance:** the Zig dep hashes live
**inside the fetched herdr source** (in the vendored `build.zig.zon.nix`), so
they self-update when `src` advances. Bumping only requires rewriting
`version` + the `src` hash via `passthru.updateScript = nix-update-script {}`.
nixpkgs' `fetchDeps` form, by contrast, needs a manual zigDeps hash bump on
every release. Cargo uses `cargoLock.lockFile = "${src}/Cargo.lock"` (no
`cargoHash` to maintain); add `cargoLock.outputHashes` only if a future herdr
release introduces non-crates.io git deps.

- New directory `pkgs/herdr/` with two files: `default.nix` (`callPackage ./package.nix {}`) and `package.nix`

- `pkgs/default.nix` auto-discovers any subdir with a `default.nix`, so no edits
  to `pkgs/default.nix` were needed — adding `pkgs/herdr/` exposed it as
  `auxera.herdr` via the overlay our dotfiles already apply (no `lib/default.nix` change).
- Build inputs: `git`, `pkg-config`, `zig_0_15`, `zstd`; on Darwin `cctools` +
  `xcbuild` (guarded by `stdenv.hostPlatform.isDarwin`, matching upstream).
  `doCheck = false` (matches upstream + nixpkgs). `meta.platforms = linux ++ darwin`.
- `env.LIBGHOSTTY_VT_ZIG_SYSTEM_DIR = zigDeps`, `ZIG`, and the `preBuild`
  `ZIG_*_CACHE_DIR` setup mirror herdr's `nix/package.nix` exactly. The
  `build.zig.zon.nix` build is sandbox-safe: every artifact is a fixed-output
  `fetchzip`/`fetchurl`/`fetchgit`, and `unpackZigArtifact` only runs `zig fetch`
  on an already-local store path (no network at build time).

**Pin in dotfiles:** `programs.herdr.package = auxera.herdr` in the `herdr.nix`
wrapper so the Auxera from-source build wins over nixpkgs' default (overlay
applied first in `lib/default.nix`).

### 0.b Config — use upstream Home Manager `programs.herdr` module

- Home Manager's `programs.herdr` module (already in our `home-manager` input)
> writes `$XDG_CONFIG_HOME/herdr/config.toml` from `programs.herdr.settings`
> (a TOML-typed attrset) and installs `cfg.package` (default `pkgs.herdr`).
- Our `herdr.nix` wrapper overrides `package` to `auxera.herdr` so the overlay
> wins.
- No `pkgs.formats.toml` / `xdg.configFile` of our own — upstream HM module does it.
- Thin wrapper responsibilities:
  1. Expose `myModules.home.herdr.enable` (matches our per-tool convention).
  2. Pin `programs.herdr.package = auxera.herdr`.
  3. Hold curated `settings` (theme, prefix, terminal defaults, toast/sound,
     remote, worktree dir, agent restore) as a Nix option; per-host `home.nix`
     can override.

Module sketch:

```nix
# modules/home/herdr.nix  (dotfiles wrapper)
{ config, lib, pkgs, inputs, ... }:
let
  cfg = config.myModules.home.herdr;
  # auxera is the Auxera nixpkgs overlay, provided through our
  # inputs.auxera.homeManagerModules.default / overlay (already wired in lib/default.nix).
  auxera = pkgs.auxera; # from auxera.overlay applied in lib/default.nix
in {
  options.myModules.home.herdr = {
    enable = lib.mkEnableOption "herdr terminal workspace manager";

    settings = lib.mkOption {
      type = (pkgs.formats.toml {}).type; # same TOML type as programs.herdr.settings
      default = {};
      description = ''
        Settings passed to programs.herdr.settings (written to
        ~/.config/herdr/config.toml). See https://herdr.dev/docs/config-reference/.
      '';
    };
  };

  config = lib.mkIf cfg.enable {
    programs.herdr = {
      enable = true;
      package = auxera.herdr; # Auxera's from-source herdr beats nixpkgs default
      settings = cfg.settings;
    };
  };
}
```

How `auxera.herdr` becomes available without changes to `lib/default.nix`:
`pkgs.auxera` is already populated by `auxera.overlays.default` (applied in
`lib/default.nix`), and `pkgs/default.nix` in `Auxera/nixpkgs` auto-discovers
new `pkgs/<name>/default.nix` files as `auxera.<name>`. So **adding
`pkgs/herdr/{default,package}.nix` in `Auxera/nixpkgs` is the ONLY change
needed** to get the package; the dotfiles `herdr.nix` wrapper then pins it.

Per-host / common defaults then live in `hosts/common/home.nix` (or our
wrapper's `config`):

```nix
myModules.home.herdr.settings = {
  onboarding = false;
  theme.name = "catppuccin";          # resolve macchiato match, see §5
  keys.prefix = "ctrl+space";          # match our tmux prefix (decision §11.3)
  terminal = { default_shell = null; shell_mode = "auto"; new_cwd = "follow"; };
  ui = { confirm_close = true; pane_borders = true; mouse_capture = true;
         toast.delivery = "herdr"; sound.enabled = true; };
  remote.manage_ssh_config = true;
  session.resume_agents_on_restore = true;
  worktrees.directory = "~/.wt-worktrees"; # share wt's store, see §8
};
```

Note: `programs.herdr.settings` is free-form TOML; pass only keys we want to
set (unset keys use herdr defaults). Validate the assembled file with
`herdr --default-config` diff once installed.

This finding **removes the risk** of maintaining a bespoke config generator and
means Phase 0 of the rollout is now near-zero-effort.

## 1. Goal and scope

Make **herdr** (`pkgs.herdr`) available across all home-manager targets as an
alternative terminal workspace manager to tmux, while keeping tmux itself
available during a transition period (not a hard cut-over).

Key principle: **add herdr alongside tmux and make the two toggle-able per
host**, rather than deleting tmux on day one. This lets us validate herdr on the
WSL devbox first, then propagate.

herdr is a drop-in *replacement*, not a wrapper around tmux. Many tmux concepts
map directly, but the sessionizer and worktree tooling needs rethinking because
herdr's workspace model absorbs part of what those scripts do.


## 2. Concept mapping (tmux → herdr)

| tmux concept                          | herdr equivalent                                             | Notes |
|---------------------------------------|--------------------------------------------------------------|-------|
| `tmux` (program)                      | `herdr` (program)                                            | No sockets to manage; `herdr` auto-attaches to default session |
| Session                               | **Workspace** (project container) or **named session**      | One workspace per repo/task is the intended pattern |
| Window                                | **Tab**                                                      | Addressable via CLI/socket API |
| Pane                                  | **Pane** (real terminal)                                    | Same idea, mouse-native |
| Prefix `C-Space`                      | `[keys] prefix` (default `ctrl+b`)                           | Can rebind to keep `ctrl+space` if desired |
| `vim-tmux-navigator` plugin           | Built-in `prefix+h/j/k/l` pane focus + navigate mode         | **Gap**: no documented neovim<->herdr pane-sync plugin (see §6) |
| `resurrect`/`continuum` plugins       | Built-in session snapshot restore (workspaces/tabs/cwd/layout)| Off by default restores structure without pane contents; pane history is opt-in experimental |
| Status bar (manual catppuccin config)  | `[theme] name = "catppuccin"` + optional `[theme.custom]`     | Built-in catppuccin theme; macchiato match may need custom override |
| `tmux-sessionizer` (fzf project pick) | herdr workspace model + a thin **herdr-sessionizer** wrapper  | herdr "from a project dir" auto-creates a workspace (see §7) |
| `wt` fish function (worktree → tmux)  | herdr worktree focus via `herdr session attach`/`--cwd`      | Built-in worktree support from sidebar (see §8) |
| `$TMUX` env detection                 | No direct equivalent; herdr has its own socket/session IDs   | Scripts must branch on which multiplexer is active |

herdr extras we gain "for free": agent detection sidebar
(`working/blocked/done/idle`) for `pi`, `opencode`, `claude`, `codex`; toast/sound
notifications; native agent session restore; `herdr --remote` SSH attach.


## 3. Inventory of tmux touchpoints in the dotfiles

Every file that references tmux and how it will need to adapt:

### 3.1 `modules/home/tmux.nix` — core tmux config
- `programs.tmux` with: prefix `C-Space`, `sensibleOnTop=false`, vim-tmux-navigator,
  resurrect, continuum, manual catppuccin status bar, terminal/ghostty truecolor
  overrides, `bind f → tmux-sessionizer`.
- **Adapter plan:** Create a parallel `modules/home/herdr.nix` that deploys
  `~/.config/herdr/config.toml` (via `xdg.configFile` or `home.file`) replicating
  the equivalent settings. Keep `tmux.nix` intact during transition. Both can
  coexist; only one runs at a time in a given terminal.

### 3.2 `modules/home/tmux-sessionizer.nix` — fzf project/worktree switcher
- A large shell script tightly coupled to tmux (`$TMUX`, `tmux list-sessions`,
  `attach-session`, `switch-client`, `new-session`, `split-window`, `send-keys`,
  pane cache keyed by tmux `pane_id`).
- **Adapter plan:** tmux-sessionizer cannot be reused as-is under herdr. See §7
  for a `herdr-sessionizer` design that reuses the repo/worktree *discovery* logic
  but swaps the session-creation backend from tmux commands to `herdr`.

### 3.3 `modules/home/wt.nix` — fish `wt` worktree manager (fish functions)
- Two tmux-coupled helpers:
  - `__wt_rename_tmux_window` → `tmux rename-window` (best-effort, guarded by `set -q TMUX`).
  - `__wt_focus_tmux_session` → `tmux switch-client` / `attach-session` /
    `new-session -ds` based on whether inside `$TMUX`. Falls back to `cd` when
    no tmux.
- **Adapter plan:** Replace the tmux session-focus helper with a herdr-aware one
  (see §8). The worktree creation logic (`git worktree add`) is multiplexer-
  independent and stays as-is. Add detection of active multiplexer (herdr vs
  tmux) to choose the focus path.

### 3.4 `modules/home/zsh.nix` — Ctrl-F binding
- `bindkey -s ^f "tmux-sessionizer\n"`
- **Adapter plan:** generalize to a `sessionizer` command name selected by the
  active multiplexer, or bind Ctrl-F to a dispatcher that runs herdr-sessionizer
  when herdr is the active manager. See §9.

### 3.5 `modules/home/fish.nix` — Ctrl-F binding
- `bind \cf "commandline -C 0; tmux-sessionizer"`
- **Adapter plan:** same as zsh — bind to the dispatcher/sessionizer name.

### 3.6 `hosts/common/home.nix` — module enable flags
- `tmux.enable = true; tmux-sessionizer.enable = true; wt.enable = true;`
- **Adapter plan:** add `herdr.enable = true;` (and/or
  `herdr-sessionizer.enable`). Keep tmux flags during transition; flip per-host
  later. Consider a single `terminalMultiplexer = "tmux"|"herdr"|"both"` option
  (see §10).

### 3.7 `hosts/common/system.nix` — system package list
- `pkgs.tmux` in `environment.systemPackages`.
- **Adapter plan:** herdr is a home-manager (user) package, not strictly needed
  at system level. Optionally add `pkgs.herdr` here too for non-home-manager
  hosts (k3s/master/oci nodes) if we want it on servers for `herdr --remote`.
  During transition: leave tmux in place.

### 3.8 `modules/home/default.nix` — import list
- imports `./tmux.nix` and `./tmux-sessionizer.nix`.
- **Adapter plan:** add `./herdr.nix` and (if kept separate) `./herdr-sessionizer.nix`.

### 3.9 catppuccin auto-enable
- `hosts/common/home.nix` sets `catppuccin.enable`, which auto-enables
  `catppuccin.tmux`; `tmux.nix` disables it (`catppuccin.tmux.enable=false`) so
  the manual bar wins.
- **Adapter plan:** herdr uses `[theme] name = "catppuccin"` per-app; no catppuccin
  nix module interaction. No change to catppuccin home module needed, but decide
  whether to pin herdr to macchiato flavor via `[theme.custom]` (see §5).


## 4. New module: `modules/home/herdr.nix` (proposed shape)

See §0 — we **delegate to the upstream Home Manager `programs.herdr` module**
rather than writing our own config generator. Our `modules/home/herdr.nix` is a
thin `myModules.home.*` wrapper that forwards `enable` + curates `settings`
(see the sketch in §0). The settings content below is what we put into
`myModules.home.herdr.settings` (persisted to `~/.config/herdr/config.toml` by
the upstream module):
- `[theme] name = "catppuccin"` (+ optional `[theme.custom]` macchiato overrides).
- `[keys] prefix = "ctrl+space"` to mirror our tmux prefix (or accept herdr's
  default `ctrl+b`; **decision needed**).
- `[terminal] default_shell`, `shell_mode`, `new_cwd` to match shell defaults.
- `[ui] mouse_capture`, `pane_borders`, `pane_gaps`, `confirm_close` etc.
- `[ui.toast]` / `[ui.sound]` tuning (sound off on shared/servers).
- `[remote] manage_ssh_config = true` for `herdr --remote` workflows.
- Maybe `[session] resume_agents_on_restore = true` (default on) — relevant since
  we run `pi` and `opencode` as agents.

No `pkgs.formats.toml` / `xdg.configFile` of our own — upstream HM module already
handles that. Validate the resulting file with `herdr --default-config` diff
once installed.

**Coexistence note:** herdr and tmux read different config files, and only one
multiplexer should be running per terminal. Stale `$TMUX` checks in scripts must
grow a parallel herdr-detection path (or use config-driven default — see §9).

## 6. Neovim ↔ pane navigation gap (open question)

- tmux side uses `vim-tmux-navigator` so `Ctrl-h/j/k/l` seamlessly moves between
  neovim splits and tmux panes.
- herdr pane focus is `prefix+h/j/k/l` (prefix-gated) and navigate mode. herdr
  does **not** document a vim-tmux-navigator-equivalent that lets neovim send a
  "leave editor → focus adjacent herdr pane" signal the way the tmux plugin does.
- **Decision needed**: are we OK losing seamless Ctrl-h/j/k/l split crossing? If
  not, investigate whether herdr exposes a socket/pane command neovim could call
  (e.g., via a neovim keybinding running `herdr pane focus --dir left`). File as
  research subtask before committing herdr as the default.


## 7. `herdr-sessionizer` design (replaces tmux-sessionizer for herdr)

Reuse the *discovery* half of `tmux-sessionizer.nix` (search-path scanning,
git-repo + worktree enumeration via `git worktree list --porcelain`,
`is_managed_worktree_dir`, `session_name_for_path`), and swap the *backend*
half:
- Instead of `tmux new-session -ds <name> -c <path>` → `herdr session attach <name>`
  or `herdr --cwd <path>` / open a workspace.
- Instead of listing `[TMUX] <session>` entries → `herdr session list`.
- Instead of `tmux switch-client` / `attach-session` → `herdr session attach`.
- Pane cache keyed by tmux `pane_id` becomes irrelevant under herdr (herdr owns
  pane IDs/session state); drop or replace with herdr pane IDs.

Recommended approach: create `modules/home/herdr-sessionizer.nix` parallel to
`tmux-sessionizer.nix`, sharing a common Nix lib for path discovery if it grows
duplicated. Bind Ctrl-F to whichever sessionizer matches the active multiplexer.

Alternative (lower effort, evaluate first): since herdr's workspace model makes
"one workspace per project" trivial and worktrees are managed from the sidebar,
we may not need an fzf picker at all — `herdr` run from a project dir already
launches/attaches that project. Decide whether the fzf flow still earns its keep
or whether we lean on `herdr --cwd` + worktree sidebar actions.


## 8. `wt` (worktree manager) adaptation

`wt` is the most behavior-dependent touchpoint. Plan:
- Keep `git worktree add/remove` paths unchanged (multiplexer-agnostic).
- Add a dispatcher in `__wt_open` / `__wt_remove`:
  - Detect active manager: prefer checking for herdr session first, else `$TMUX`,
    else plain `cd`.
  - herdr path: open the worktree as a **workspace** —
    `herdr session attach <repo>/<worktree> --cwd <target>` or the sidebar's
    "Open worktree" (herdr can group worktrees under a parent workspace; see
    `[worktrees] directory` config). Set `[worktrees] directory` to match our
    existing `~/.wt-worktrees` so herdr-native worktree creation lands in the same
    place as `wt`.
  - Replace `__wt_rename_tmux_window` with `herdr` pane/tab rename (or drop it —
    herdr labels workspaces from their cwd/path automatically).
- Make the fish helpers degrade gracefully (already do for tmux) so `wt` still
  works with no multiplexer at all.

**Open question:** herdr's `[worktrees] directory` (`~/.herdr/worktrees` default)
should be set to `~/.wt-worktrees` to avoid a second worktree store. Confirm
herdr's checkout layout (`<directory>/<repo>/<branch-slug>`) is compatible with
`wt`'s `repo_dir/name` layout, or accept that herdr-created and `wt`-created
worktrees live under the same root with slightly different naming. This affects
`__wt_repo_dir` assumptions.


## 9. Shell binding dispatcher (zsh + fish)

Add a small dispatcher instead of hardcoding `tmux-sessionizer`:
- Determine the active multiplexer (herdr if a herdr session exists/attached,
  else tmux if `$TMUX`, else pick the system default).
- Run the matching sessionizer (`herdr-sessionizer` vs `tmux-sessionizer`).

Implementation detail: detecting "inside herdr" from a shell init isn't as clean
as checking `$TMUX`. herdr sets `HERDR_*` env vars on custom commands but not
necessarily on plain panes; verify what env herdr sets in a pane shell (per
`herdr --help` / define a sentinel). If unreliable, fall back to a config-driven
choice via a Nix option (`terminalMultiplexer`) rather than runtime detection.

Pragmatic recommendation: **config-driven** (a Nix option selects default
manager; the Ctrl-F binding calls that manager's sessionizer). Simpler and
deterministic; runtime detection only needed to *not* launch the wrong
multiplexer's sessionizer when the other is already running.


## 10. Rollout strategy (phased, reversible)

1. **Phase 0a — Add herdr to `Auxera/nixpkgs` as a from-source package.**
   - Create `pkgs/herdr/default.nix` + `pkgs/herdr/package.nix` (mirror
     `pkgs/opencode`'s two-file split).
   - Build from source using the same tooling as the remote herdr repo's own
     `nix/package.nix` (Rust + vendored `libghostty-vt` Zig build via `zon2nix`-
     generated `build.zig.zon.nix`), OR model on nixpkgs' `by-name` derivation
     (`fetchFromGitHub` + `buildRustPackage` + `zig_0_15.fetchDeps`) for proven
     hash parity — both are from-source. **Pick one (see §11.8).**
   - `pkgs/default.nix` auto-discovers it → `auxera.herdr` via the overlay our
     dotfiles already apply. No changes to `lib/default.nix`.
   - Verify it builds on the Auxera systems list; add `aarch64-linux` to
     `Auxera/nixpkgs`' `systems` if needed (our oci nodes use it).
   - Bump the Auxera flake input in dotfiles + rebuild; `nix run .#herdr --version`.

2. **Phase 0b — Add herdr config via upstream HM module (no behavioral change in dotfiles).**
   - Add `./herdr.nix` (thin wrapper) to `modules/home/default.nix`, default
     `enable=false`. The wrapper pins `programs.herdr.package = auxera.herdr`
     (so the from-source Auxera build wins over any nixpkgs default) and
     forwards `settings`.
   - Curated `settings` live in the wrapper's defaults / `hosts/common/home.nix`
     (§0 sketch).
   - Rebuild; verify `herdr` launches on wsl-devbox. (Config side is trivial
     because we lean on the upstream `programs.herdr` module — no bespoke config
     generator to write or maintain.)

3. **Phase 1 — Enable on wsl-devbox only, tmux still default.**
   - `hosts/wsl-devbox/home.nix`: set `herdr.enable = true`.
